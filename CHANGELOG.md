## [Unreleased]

### Added

- `src/ordo_ai/nodes/intent.py` adds `_regex_rescue(raw)`: on JSON parse failure, scans the raw model output for the first substring matching a `_VALID_LABELS` name and the first float/int-like token for confidence (default `0.9` if none found), used as a fallback in `predict_intent()` before giving up to `chitchat_oos`/`0.0`. `_SYSTEM_PROMPT` gains a one-shot example output line to steer the model toward strict JSON.
- `data/menu.json`: expands `description` fields for all 11 items into longer natural-language Indonesian text (was terse fragments; `Es Teh Manis` previously had `description: null`), improving `_llm_inquiry_response()`'s input context.
- `src/ordo_ai/tools/menu.py`: `find_menu_items(name, threshold=0.55, band=0.02)` returns all menu items within `band` of the top fuzzy-match score (instead of a single best match), filtered to require the query's words be a subset of the candidate name's words (blocks cross-word false positives like `"nasi goreng"` matching `"Ayam Goreng"`); falls back to the single best match if no candidate passes the word-subset filter. `src/ordo_ai/state/schemas.py` adds `PendingItem(TypedDict)` (`name`/`quantity`/`notes`/`candidates`) and `OrderState.pending_item`. `src/ordo_ai/nodes/order_agent.py`'s `run()` calls `find_menu_items` instead of `find_menu_item` for non-remove intents; when it returns >1 candidate, sets `state["pending_item"]` and `needs_clarification=True` with a numbered-options message instead of adding to cart. `src/ordo_ai/nodes/dialog_agent.py`'s `run()` resolves a pending `confirm` by parsing the user's picked number via `_pick_candidate_index(text)` (matches Indonesian ordinal words `pertama`/`kedua`/etc., number words `satu`/`dua`/etc., or a bare digit) against `pending_item["candidates"]`, then calls `tools.cart.add_item(cart, menu_item, quantity, notes)`; re-prompts with the option list if no valid pick is found.
- `src/ordo_ai/graphs/main_graph.py`: `_timed(name, fn)` wraps every graph node to record per-node latency into `state["node_timings"][name]` (also logged via `logger.debug`) using `time.perf_counter()`; `build_graph()` wraps all ten nodes (`stt`, `normalize`, `disfluency`, `ner`, `intent`, `clarify`, `order_agent`, `menu_agent`, `dialog_agent`, `fallback_agent`) with it. `OrderState` gains `node_timings: dict[str, float]`.
- `src/ordo_ai/nodes/menu_agent.py`: `run()` now branches on `state["intent"] == "menu_inquiry"` — inquiry responses list descriptions and skip auto-adding to cart; other intents keep the prior price-list + `add_item` behavior.
- `src/ordo_ai/nodes/intent.py` adds `order_swap` to `_VALID_LABELS`/`_LABEL_DESCRIPTIONS` ("user wants to replace/swap one item in their order with another, e.g. 'ganti X ke Y'"), routed to `order_agent` in `src/ordo_ai/nodes/router.py`'s `_INTENT_TO_AGENT`. `src/ordo_ai/nodes/order_agent.py`'s `run()` handles it by removing `parsed_items[0]` via `find_cart_index_fuzzy` and adding `parsed_items[1]` via `find_menu_items`; if the add side is ambiguous (>1 candidate) it stores `remove_name` on `PendingItem` (new field, `src/ordo_ai/state/schemas.py`) so the eventual disambiguation answer still removes the original item before adding the picked one. New `_resolve_pending(state)` in `order_agent.py` resolves an active disambiguation from entity spans (`DISH`/`DRINK`) before falling through to normal intent handling, mirroring the same swap-aware removal. `src/ordo_ai/nodes/dialog_agent.py`'s pending-item resolution also pops `remove_name` from the cart via `find_cart_index` before calling `add_item`, and falls back to substring-matching the candidate name against `repaired_text` when `_pick_candidate_index` finds no ordinal/digit.
- `src/ordo_ai/tools/cart.py` adds `find_cart_index_fuzzy(cart, query, threshold=0.55)`: `SequenceMatcher` ratio match against cart item names, requiring `query`'s words be a subset of the item name's words (blocks cross-word false positives) and boosting (+0.2) exact word-subset matches; used by `order_agent.py`'s `order_remove_item`/`order_swap` handling instead of exact `find_menu_item` + `find_cart_index` lookups.
- `src/ordo_ai/nodes/menu_agent.py` adds `_llm_inquiry_response(user_question, items)`: builds a menu-info block and prompts `mlx-community/Qwen2.5-7B-Instruct-4bit` (via `intent._load()`, reused rather than loading a second model) with `_INQUIRY_SYSTEM` to generate a natural-language Indonesian answer (`max_tokens=128`), replacing the previous static `"{name} - Rp{price}: {description}"` listing for `menu_inquiry` responses.
- `scripts/run_voice_session.py`'s `on_final()` prints per-node latency after each turn: `timings = result.get("node_timings", {})`, formatted as `k=v.4fs` pairs; the per-node `[{node_name}]` debug print now excludes `node_timings` from the displayed dict to avoid clutter.
- `scripts/run_voice_session.py`: adds module-level `session_state: dict` carried across utterances — `on_final(text)` now seeds each graph invocation with `{k: v for k, v in session_state.items() if v is not None}` merged with the new `raw_text`, and after each run copies `_SESSION_CARRY_KEYS = ("cart", "pending_item", "needs_clarification")` from the result back into `session_state`, so cart contents and pending clarification survive between wake-word turns instead of resetting every utterance. Also prints `cart={session_state.get('cart', [])}` after each non-clarification response.

### Fixed

- `src/ordo_ai/nodes/dialog_agent.py`'s `run()`: guards pending-item resolution with `state.get("needs_clarification")` in addition to `pending.get("candidates")`, so a stale `pending_item` left in `session_state` from a prior turn no longer gets reinterpreted as a disambiguation answer on an unrelated later turn. The re-ask branch now sets `clarification_message` (previously only `agent_response`), matching what `scripts/run_voice_session.py`'s `on_final()` reads.
- `scripts/run_voice_session.py`'s `on_final()`: falls back to `result.get("agent_response")` when `clarification_message` is absent, instead of a `KeyError` on `result['clarification_message']`.
- `src/ordo_ai/nodes/intent.py`'s `predict_intent()`: handles the MLX Qwen model returning a `{"label": score}` shorthand reply (no `"intent"`/`"confidence"` keys) by picking the highest-scoring key present in `_VALID_LABELS`, instead of raising and falling back to `chitchat_oos`. `run()` now classifies `state["normalized_text"]` instead of `state["repaired_text"]`.

- `notebooks/6.1_intent_classification.ipynb`: fine-tunes `indobenchmark/indobert-base-p1` (`AutoModelForSequenceClassification`, `p1` over `p2` for faster inference since intent runs on every utterance) on `text_normalized` from `data/normalized/intent_dataset_normalized.jsonl` (2060 records, 10-way intent labels already present, no labeling stage needed). `train_test_split` (scikit-learn, stratified by `intent`, `random_state=42`) splits 80/10/10 into `data/intent/{train,val,test}.jsonl` + `label_map.json`. Plain unweighted `Trainer` (`learning_rate=2e-5`, `warmup_ratio=0.1` linear, 5 epochs, batch size 16, `load_best_model_at_end` on macro-F1) — imbalance here is mild (1.3x max: disfluency-augmented `order_create` examples from `3.1` land at 260 vs 200/class elsewhere), unlike the weighted-loss disfluency/NER stages. `compute_metrics` reports accuracy + macro-F1 via `evaluate`; test-set eval adds per-intent `classification_report` and prints up to 15 misclassified examples. Saves to `models/indobert-intent-final`. `predict_intent(text)` reloads the saved model standalone and returns predicted intent plus per-class softmax probabilities. Executed: test loss=0.0141, accuracy=0.995, macro-F1=0.995; per-intent `classification_report` shows 8/10 intents at perfect 1.000 f1, `order_modify_quantity` f1=0.976 (n=20), `order_remove_item` f1=0.974 (n=20, recall=0.950).
- `notebooks/3.1_disfluency_bio_labeling.ipynb`: rule-based BIO labeler over `data/normalized/intent_dataset_normalized.jsonl`'s `text` field (disfluency markers are preserved there; `text_normalized` only strips punctuation, so neither field is a ready-made label) using a 5-tag taxonomy: `IP` (filler), `FS` (false start), `RC` (repeat), `RP`/`RM` (reparandum/repair pair). `label_core(toks, raw_toks)` does a single left-to-right pass tagging `FS` (hyphen-truncated token immediately followed by its completed form, e.g. `na- nasi`), `RC` (exact adjacent word duplicate, excluding `VOCATIVES` like `kak`/`bang`/`mas` so direct address never misfires as a repeat), and standalone `IP` fillers (`eh`, `anu`, `um`, `hmm`, `ee`, `em`). `apply_repair_bridges(toks, raw_toks, labels)` runs a second pass restricted to `", <bridge>,"` patterns where both flanking tokens are numerals with different values (e.g. `dua, eh, tiga`) — corpus inspection showed `anu`-bridges and most `eh`-bridges are pure mid-clause hesitation with no actual correction, so unrestricted bridge detection produced false `RP`/`RM` tags on cancel/query sentences (e.g. `mbak, anu, mie goreng...dibatalin` wrongly tagging `mbak` as `RP`); restricting to numeral swaps leaves only the 2 genuine repairs in the corpus. `validate_bio` auto-corrects orphan `I-` tags to `B-` (guards against future heuristic changes, though current logic never produces orphans). Labels align to `LazarusNLP/NusaBERT-base` (IndoBERT continued-pretrained on Indonesian + regional languages incl. Javanese — chosen over plain IndoBERT for Ordo's Javanese-accented speech) WordPiece tokens via `align_labels_with_tokens` (`word_ids()`-based, first-subword convention, `max_length=64`, `-100` for continuations/specials). `dominant_type(labels)` picks each record's rarest present tag (priority `FS > RP > RM > RC > IP > fluent`) for stratification; `train_test_split` runs twice (80/20, then the 20% into 10/10), so classes need >=4 occurrences to survive both splits — `MIN_STRATIFY_COUNT = 4` forces `FS`(3)/`RP`(2)/`RM`(2) entirely into train rather than crashing on a 1-member class. Writes `data/disfluency/{train,val,test}.jsonl` (1601/199/200) and `label_map.json` (7 labels: `O`, `B-IP`, `B-FS`, `B-RC`, `I-RC`, `B-RP`, `B-RM`).
- `notebooks/5.1_disfluency_repair_seq2seq.ipynb`: fine-tunes `Wikidepia/IndoT5-small` (`AutoModelForSeq2SeqLM`) on `data/disfluency_repair/{train,val,test}.jsonl` (`text -> text_normalized` pairs), treating disfluency repair as text-to-text generation rather than BIO span tagging, since some repairs are replacements (self-corrections like "dua, eh, tiga") not pure deletions. Uses `Seq2SeqTrainer`/`Seq2SeqTrainingArguments` with `DataCollatorForSeq2Seq`, `MAX_INPUT_LEN`/`MAX_TARGET_LEN=64`, runs on `mps`. Saves to `models/disfluency_repair_indot5_small`.
- `scripts/split_disfluency_repair_dataset.py`: builds `data/disfluency_repair/{train,val,test}.jsonl` from `data/normalized/intent_dataset_normalized.jsonl`. `to_pair(row)` maps each record to `{id, input: text, target: text_normalized, intent}`; `train_test_split` (scikit-learn, `random_state=42`) splits 80/10/10 stratified by `intent` to keep the 10-way intent balance across splits.
- `notebooks/4.2_ner_training.ipynb`: full IndoBERT entity-NER training pipeline mirroring `3.2_bio_labeling.ipynb`'s structure. `compute_label_counts`/inverse-frequency `class_weights` feed a `WeightedTrainer(Trainer)` (overridden `compute_loss`) fine-tuning `AutoModelForTokenClassification.from_pretrained("indobenchmark/indobert-base-p1", num_labels=19, ...)` over the 9-tag entity schema (18 labels + `O`). `TrainingArguments`: AdamW, `learning_rate=2e-5`, `warmup_ratio=0.1` linear schedule, `weight_decay=0.01`, 5 epochs, batch size 16, eval/save per epoch, `load_best_model_at_end` on val F1. `compute_metrics` scores entity-level precision/recall/F1/accuracy via `seqeval` after stripping `-100` positions. Saves to `models/indobert-ner-bio-final`. `trainer.predict(test_dataset)` reports per-entity-type breakdown on the held-out test split (rare tags `REMOVE`/`TABLE` flagged as most likely weak). `predict_entities(text)` reloads the saved model standalone, tokenizes raw text, decodes first-subword predictions back to word-level BIO tags, and extracts contiguous `B-X`/`I-X` spans as `{label, text, start, end}` — the structured order format downstream LangGraph nodes will consume.
- `notebooks/4.1_ner_labeling.ipynb`: entity-NER labeling pipeline using a local LLM (`mlx-community/Qwen2.5-7B-Instruct-4bit` via `mlx_lm.load`/`generate`) instead of rule-based span matching, since dish/drink names in the dataset go beyond `data/menu.json`'s 11 items. `label_sentence(text, max_retries=2)` prompts the model with a fixed 9-tag schema (`DISH`, `QUANTITY`, `MODIFIER`, `SIZE`, `DRINK`, `ADD_ON`, `REMOVE`, `TABLE`, `TAKE_AWAY`) and `parse_entities(raw)` validates the JSON span-list response, retrying on parse failure.
